## ADDED Requirements

### Requirement: Dedicated Web entrypoint
The system SHALL provide a minimal Web entrypoint under `web/` and serve its build output as Worker assets.

#### Scenario: Web build output served
- **WHEN** the Web app is built
- **THEN** it outputs static assets to `web/dist` for the Cloudflare Worker assets binding

#### Scenario: Worker preserves API routes
- **WHEN** a request targets `/api`, `/mcp`, `/oauth`, `/auth`, Stripe webhook, or WeChat callback routes
- **THEN** the Worker handles those routes before SPA asset fallback

### Requirement: Astryx design system
The Web entrypoint SHALL use Astryx as the required design system.

#### Scenario: Astryx providers mounted
- **WHEN** the Web app renders
- **THEN** it wraps the app with Astryx `ThemeProvider` and `LinkProvider`

#### Scenario: Astryx CSS layer order
- **WHEN** the Web app imports CSS
- **THEN** it imports Astryx reset, Astryx core CSS, and project theme CSS in the required layer order

#### Scenario: Tailwind UI not used
- **WHEN** new Web pages or components are implemented
- **THEN** they use Astryx components/theme CSS rather than Tailwind UI classes as the styling system

### Requirement: TanStack routing
The Web entrypoint SHALL use TanStack Router file-based routing generated by the Vite plugin.

#### Scenario: File routes generated
- **WHEN** the Web app runs in development or build
- **THEN** the TanStack Router Vite plugin generates the route tree from file routes

#### Scenario: React Router removed
- **WHEN** TanStack Router is adopted
- **THEN** React Router dependencies and route code are removed from the Web entrypoint

#### Scenario: Astryx LinkProvider uses AppLink
- **WHEN** Astryx components render internal links
- **THEN** a central `AppLink` adapter maps Astryx link props to TanStack Router `Link`

### Requirement: Web server state and forms
The Web entrypoint SHALL use TanStack Query for server state and native forms with Zod validation for onboarding forms.

#### Scenario: Server state fetched through Query
- **WHEN** Web pages load `/me`, onboarding status, account, billing, MCP config, or session data
- **THEN** they use TanStack Query with explicit cache invalidation for mutations

#### Scenario: Native form validation
- **WHEN** an Operator submits login or onboarding forms
- **THEN** the Web validates input with Zod and Astryx form components before or alongside backend validation

### Requirement: Web route inventory
The Web entrypoint SHALL include the first-release login, onboarding, billing, MCP config, security sessions, Terms, and Privacy routes.

#### Scenario: Protected route redirects
- **WHEN** an unauthenticated browser requests a protected Web route
- **THEN** the system redirects to login with a `returnTo` target

#### Scenario: Login returns to target
- **WHEN** an Operator completes login with a `returnTo` target
- **THEN** the system returns the browser to that target if it is safe

### Requirement: Web login UX
The Web login UI SHALL present email code login before GitHub login.

#### Scenario: Email login primary
- **WHEN** an unauthenticated person opens the login page
- **THEN** the email-code path appears as the primary login path and GitHub appears as an alternative

### Requirement: Anti-AI-slop Web acceptance
The Web entrypoint SHALL comply with `docs/design/anti-ai-slop-rules.md`.

#### Scenario: Astryx dense docs required
- **WHEN** an implementation branch modifies Web pages or components
- **THEN** it runs Astryx dense docs for principles, component list, theme, and used components before implementation

#### Scenario: Screenshot review required
- **WHEN** critical Web pages are implemented or materially changed
- **THEN** the implementation provides screenshots for anti-slop visual review

#### Scenario: Chinese copy reviewed
- **WHEN** Web or CLI Chinese copy is added
- **THEN** the copy is reviewed for concrete operational language and absence of generic SaaS filler

### Requirement: Web tests
The Web entrypoint SHALL include tests for API/viewmodel boundaries, route guard behavior, and critical page smoke rendering.

#### Scenario: API response drift caught
- **WHEN** a key Web API response shape changes unexpectedly
- **THEN** shared types and Zod boundary tests fail before production deployment
